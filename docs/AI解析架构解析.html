ARCHITECTURE

AI 地址解析架构解析

规则引擎 + AI 兜底的双层架构，如何用「精准匹配 + 常识推理」破解中文地址解析难题
● 模块: address_resolver_v2 ● 引擎: Trie + Levenshtein + DeepSeek ● 更新: v2.6

01问题背景：为什么地址解析这么难

中文地址解析的本质，是把一段非结构化文本切成 省 / 市 / 区县 / 街道 四级结构，并匹配到国标 6 位区划代码。难点在于：

🧩 输入不规范
全角半角混用、空格缺失、省市省略、简称代称（"沪""京""经开区"）、街道后缀截断（"西湖路"撞"西湖区"）
🔀 歧义丛生
同名区县（全国多个"城区""市中区"）、单字简称撞字（"单"→单县）、县市同名（"伊宁县" vs "伊宁市"）
📊 数据层级深
3000+ 区县、自治县民族后缀冗长、直辖市下辖区归属特殊、无下辖区市（东莞/中山）需特殊处理
🎯 准确率与成本平衡
纯规则覆盖不全，纯 AI 成本高、延迟大。需要"规则优先、AI 兜底"的混合策略

02总体架构图谱

整个引擎采用 "规则解析 → 阈值判定 → AI 兜底" 的三段式流水线。规则引擎先给出 1~3 个候选与置信度，当置信度不足或存在歧义时，再交由 AI 在候选基础上做最终裁决。

03规则解析：核心技巧与机制

双 Trie 索引 全称/简称双轨 层级递进匹配 父子关系约束 置信度惩罚体系 Levenshtein 模糊

3.1 数据层：双 Trie 索引 + 别名字典

启动时把全国 3000+ 区划构建两棵前缀树（Trie），实现 O(n) 文本扫描下精确召回所有可能匹配：

full_trie   # 全称：「北京市」「静安区」「恩施土家族苗族自治州」
short_trie  # 简称/别名：「北京」「静安」「恩施」「经开区」「浦东」
            # 含 PROVINCE_ALIAS（沪/京/桂）+ 后缀剥离（"自治区"→"广西"）
            # + 自治县正则提取（"广西大化瑶族自治县"→"大化"）
Trie 的 search_from(text, start) 沿字符路径下行，每个有 data 的节点都收集为命中。一棵树同时返回省、市、区三级命中，避免多次扫描。

3.2 匹配流水线：层级递进 + 双轨合并

规则按 省 → 市 → 区县 顺序匹配，每一级同时跑全称与简称，再合并去重。关键机制：

机制作用示例
_merge_full_short全称优先，简称不与全称重叠才采用"北京市"已匹配，则"北京"简称不再生成候选
_match_city 父子过滤有省级命中时，市必须属于该省且位置在省后避免"浙江省南昌市"误组合
_is_short_invalid简称后紧跟"区县市"等后缀时校验是否构成全称"北京" + "市" = "北京市" → 不再算简称
_is_county_short_street_conflict区县简称后紧跟街道后缀时丢弃，防撞字"西湖" + "路" → 不匹配"西湖区"
单字省份简称黑名单单独一个字（沪/京/渝）容易被误命中"上海"匹配全称，不依赖单字"沪"
infer 反推匹配到下级时反查父级 code 填充上级只写"西湖区"→自动反推浙江省杭州市

3.3 置信度评分：多层惩罚体系

规则给每个候选打 0~100 分，分数高低决定是否触发 AI。评分由 base + level_bonus + 多项惩罚 组成：

base        : 全称=100 / 简称=80 / infer=70 / fuzzy=60 / 无匹配=50
level_bonus : 省市区完整 +20 / 省市完整 +10 / 仅省或市 +5
stability   : 候选差距小（歧义）-20
village_penalty     : 街道残留区划后缀 -25，街道被截断 -25
same_name_penalty   : 同名区县 -15
building_conflict   : 同名 + 建筑名（如医院/学校）-10
single_char_short   : 单字区县简称 -25
truncation_penalty  : 检测到截断匹配 -30
county_missing      : 区县缺失 -10~-20，且置 need_ai_review=true
任何一项惩罚触发都会同时把 need_ai_review 置为 true，强制进入 AI 复核路径。这是规则"知道自己不行"的关键自检机制。

3.4 cityCode 引导：业务上下文加权

调用方传入投保地市 city_code 时，引擎做两件事：

  1. 引导匹配：在指定地市下辖区县里直接搜全称/简称，绕过全省扫描，避免同名干扰
  2. 置信度加权：候选 city == city_code 时 +20，同省地市 +10

3.5 模糊兜底：Levenshtein + 拼音

规则零候选时启用。把地址切成 2~8 字汉字段，对每段跑：

  • 编辑距离：city ≤1，county ≤2，得分 = 60 - 距离×10
  • 拼音同音：lazy_pinyin 全拼相等，得分 50

典型场景：用户输入错别字（"哈而滨"→哈尔滨）、拼音输入（"hangzhou"→杭州音匹配）。

3.6 直辖市与无下辖区市的特殊规范化

_normalize_candidate 处理两类边界：

  • 直辖市：数据集里"北京市"既是省也是市。若 AI/规则把区填到了 city 字段，系统自动提升到 county，并把 city 改为"北京市"本身，同时给 AI 候选追加 reason 说明
  • 无下辖区市：东莞/中山/儋州/嘉峪关，自动补一个"市辖区"作为 county

04AI 解析：兜底逻辑架构

DeepSeek Chat instructor 结构化输出 Prompt 工程 LRU+TTL 缓存 峰谷计费

4.1 触发条件：什么时候才调用 AI

AI 不是默认开启，而是由 _need_ai() 判定，避免无谓成本与延迟：

def _need_ai(candidates):
    if not candidates:                              # 规则零候选
        return True
    if candidates[0].need_ai_review:                # 规则自检标记
        return True
    if candidates[0].confidence <= 75:              # 置信度阈值
        return True
    if len(candidates) >= 2 and \
       (candidates[0].confidence - candidates[1].confidence) < 10:  # 歧义 gap
        return True
    return False

4.2 Prompt 工程：强/弱候选双策略

AI 的核心技巧是 "看规则给不给力，给不同指令"。判定逻辑：

def _is_weak_rule_candidate(cand):
    # 区县在原文中精确匹配（full/short）即视为完整候选
    if cand.county_source in {"full", "short"}:
        return False    # 强候选
    return True         # 弱候选：区县缺失或仅靠 infer/fuzzy 撞字
🟦 强候选策略
规则已精确匹配区县。Prompt 严格约束："只能从候选列表中选择代码，严禁编造不在候选中的代码"。AI 只做消歧与排序，不可越界
🟩 弱候选策略
规则仅靠撞字反推（如"灵石路"→灵石县）。Prompt 放开约束："候选仅为参考，可能为误匹配，应基于常识从全国真实区划中选择"

4.3 上下文注入：把规则知道的都告诉 AI

Prompt 不仅传地址，还把规则解析过程中发现的关键线索塞给 AI：

  • 候选列表：省/市/区县的 code+name，让 AI 在已知答案里选
  • 同名区县消歧候选：所有同名区县及其省市，避免 AI 凭印象乱选
  • 截断候选：原文可能含更长区划名（如"和县"实为"和平县"），列出 endswith 候选
  • cityCode 业务提示：投保地市 + 同省地市列表，"若存在歧义优先选提示地市"
  • 直辖市数据模型说明：明确告诉 AI"区填 city、county 留 null"，避免模型按国标填错

4.4 结构化输出与容错校验

用 instructor + pydantic 强制 AI 返回结构化 AIAddressResponse，杜绝 JSON 解析失败。拿到结果后做多重校验：

校验处理
code 查不到或名称不符按 name 反查（AI 常知道名称不知代码）
直辖市省级码国标 xx0000映射回数据集 xx0100（_MUNI_PROV_AI_COMPAT）
省和市不匹配丢弃较不可信一级，以有 name 支持的为准
市和县不匹配以 county 为准反推 city 与 province
三层全空丢弃该候选

4.5 合并去弱：AI 与规则的最终裁决

_merge_ai_candidates 把 AI 与规则候选按 (省,市,区) 三元组去合并，保留置信度高者。关键策略：若 AI 已给出结果，丢弃规则的"弱猜测"候选，避免规则撞字结果污染输出。

4.6 缓存与计费：成本控制双保险

📦 LRU + TTL 缓存
OrderedDict 实现，maxsize=10000、ttl=86400。Key=(address, city_code)。命中直接返回，零成本零延迟
💸 峰谷计费
DeepSeek V4 区分高峰/平时。按 prompt_cache_hit/miss + output 三类 token 分别计价，命中本地缓存不计费
peak_hours = "09:00-12:00;14:00-18:00"   # 北京时间，左闭右开
# 平时:  hit 0.2 / miss 1.0 / out 2.0  元/百万token
# 高峰:  hit 0.04 / miss 2.0 / out 4.0 元/百万token（缓存命中反而更便宜）

05AI 调用链路：组件库与实现思路

AI 兜底不是简单地"调一个 HTTP 接口"，而是用 4 个组件库 + 1 个远端服务 拼装出一条"懒加载 → 缓存 → 结构化输出 → 校验 → 计费"的完整链路。下面拆解每个组件的角色与串联方式。

5.1 组件库清单与职责

组件角色在链路中的作用
openai
OpenAI / AsyncOpenAI		 HTTP 客户端 与 DeepSeek API 通信（DeepSeek 兼容 OpenAI 协议）。负责鉴权、连接、超时、重试。base_url 指向 api.deepseek.com，api_key 从环境变量或 config 读取
instructor
结构化输出层		 输出规整器 用 instructor.from_openai(client) 包装客户端，把 pydantic schema 自动转为 function calling / tool_choice，强制模型返回符合 schema 的 JSON，杜绝解析失败
pydantic
Schema 定义		 数据契约 定义 AIAddressItem / AIAddressResponse 两个模型，字段含 Field(description=...) 作为 prompt 提示。既是给 instructor 的输出 schema，也是返回值校验器
asyncio + OrderedDict
AICache		 本地缓存 OrderedDict 实现 LRU（move_to_end + popitem(last=False)），叠加 TTL 过期。asyncio.Lock 保证并发读写安全。Key=(address, city_code)
DeepSeek API
远端 LLM		 推理引擎 实际执行地址理解的 LLM 服务。支持 deepseek-v4-flash（快）与 deepseek-v4-pro（强推理）。通过 extra_body={"thinking":{"type":"disabled"}} 关闭思考模式以兼容 instructor

5.2 组件串联链路图

下图标示了一次 AI 兜底调用的完整链路：从规则引擎判定触发 AI 开始，依次经过 缓存查询 → Prompt 构建 → instructor 包装 → OpenAI 客户端 → DeepSeek 推理，再沿虚线返回：结构化结果 → 多重校验 → 回写缓存 → 交回规则引擎合并。每个节点都标注了中文名与职责，颜色区分四类角色。

链路读图： 蓝色 AddressResolver 在规则置信度不足时调用 绿色 AIResolver；先查 橙色 AICache，命中直接返回零成本结果；未命中则由 紫色 Prompt 构建把规则候选与上下文打包成提示词，经 instructor 包装成结构化调用，由 AsyncOpenAI 发往 DeepSeek 推理；返回结果按 黄色 pydantic schema 校验后，沿虚线回写缓存并交回规则引擎做合并去弱。

5.3 instructor 结构化输出原理

原始 openai SDK 返回自由文本 JSON，需要自己 json.loads + 字段校验，模型偶尔会输出不合规 JSON 导致解析失败。instructor 的做法是把 pydantic schema 翻译成 OpenAI 的 function calling / tool_choice 协议：

# pydantic 模型
class AIAddressItem(BaseModel):
    province_code: str = Field(description="省级6位区划代码")
    province_name: str = Field(description="省级名称")
    county_code: Optional[str] = Field(default=None, ...)
    confidence: float = Field(description="本次判断的置信度0-100")
    reason: str = Field(description="选择该结果的简短理由")

class AIAddressResponse(BaseModel):
    results: list[AIAddressItem] = Field(description="按可能性从高到低排序")

# instructor 包装 + 调用
client = instructor.from_openai(AsyncOpenAI(api_key=..., base_url=...))
resp = await client.chat.completions.create(
    model="deepseek-v4-flash",
    response_model=AIAddressResponse,   # ← 关键：强制 schema
    messages=[{"role":"system",...}, {"role":"user",...}],
    temperature=0.1,
    max_retries=2,
)
# resp 直接是 AIAddressResponse 实例，无需 json.loads
instructor 在背后把 AIAddressResponse 转成 tools 参数发给模型，模型按 function calling 格式返回，instructor 再用 pydantic 校验并实例化。失败时自动重试（max_retries）。

5.4 实现思路要点

🔁 懒加载客户端
_ensure_async_client 首次调用时才 import + 实例化，避免启动耗时与无 AI 场景下的依赖缺失报错
⚡ 同步/异步双客户端
同时维护 OpenAI 与 AsyncOpenAI 两套，resolve 走同步、resolve_async 走异步，适配 FastAPI 高并发
💾 缓存前置
调用 API 前先查 AICache，命中直接返回零成本结果。写缓存用 asyncio.Lock 保证并发安全
🛡️ 异常吞掉 + 零费用兜底
整个 AI 调用包在 try/except，失败返回 [] 与 _zero_cost()，不让 AI 故障拖垮主流程
🚫 思考模式开关
通过 extra_body={"thinking":{"type":"disabled"}} 显式关闭思考模式——思考模式与 instructor 的 tool_choice 不兼容，会报 400
📊 原始响应挂载
instructor 把原始 ChatCompletion 挂在 resp._raw_response，从中取 usage 做 token 计费统计

5.5 一次完整调用的伪代码

async def resolve_async(address, text, rule_candidates, loader, city_code):
    # 1. 查缓存
    cached = await self.cache.get(address, city_code)
    if cached:
        return cached, True, zero_cost          # 命中：不计费

    # 2. 懒加载 instructor 客户端
    client = await self._ensure_async_client()

    # 3. 构建 prompt（强弱策略 + 上下文注入）
    weak = self._is_weak_rule_candidates(rule_candidates)
    system = self._build_system_prompt(city_code, weak)
    user = self._build_user_prompt(address, candidates, same_name, truncation, city_hint, weak)

    # 4. instructor 结构化调用 DeepSeek
    try:
        resp = await client.chat.completions.create(
            model=self.model,
            response_model=AIAddressResponse,
            messages=[{"role":"system","content":system},
                      {"role":"user","content":user}],
            temperature=0.1,
            max_retries=2,
            extra_body={"thinking":{"type":"disabled"}},
        )
    except Exception:
        return [], False, zero_cost              # 失败兜底

    # 5. 从原始响应取 usage 计费
    raw = getattr(resp, "_raw_response", None)
    cost = self._build_cost_info(getattr(raw, "usage", None))

    # 6. pydantic → Candidate + 多重校验
    result = [self._to_candidate(item, loader, text) for item in resp.results]

    # 7. 写缓存（异步锁保护）
    if result:
        await self.cache.put(address, city_code, result)

    return result, False, cost

06AI 解决了规则解决不了的什么问题

🧠 常识推理
"上海市南京东路"→黄浦区，"无锡市大池路督府天承"→滨湖区。规则只能撞字，AI 用语义常识推断街道所属区县
🔀 同名消歧
多个"城区""市中区"，AI 结合上下文与知名度排序，无法确定时返回所有候选并标注"同名歧义"
✂️ 截断修复
"和县"实为"和平县"、"伊宁"可能是伊宁县或伊宁市。AI 识别原文意图，补全真实区划
🔤 错别字容错
超出 Levenshtein 距离的语义级错别字（"哈而滨市"→哈尔滨），AI 凭语言理解直接纠正
🏚️ 缺失补全
地址只到省市、区县缺失时，AI 根据街道/楼盘/路名高置信度补全 county
🚫 拒绝乱猜
规则"弱猜测"撞字（"灵石路"→灵石县）会被 AI 识别并拒绝，基于常识给出真实区划

07端到端解析流程

① 输入与预处理
原始 address + 可选 city_code。全角转半角、去空格，得到归一化 text
② 规则解析 rule
双 Trie 扫描 → 省/市/区三级匹配 → 全称简称合并 → 父子约束过滤 → 组合候选 → 模糊兜底（如零候选）→ 直辖市规范化 → 置信度评分
③ cityCode 引导（可选）
在指定地市下辖区县里直接匹配，候选 city 命中则置信度 +20，同省 +10
④ AI 触发判定
零候选 / need_ai_review / confidence≤75 / 歧义 gap<10 → 触发 AI；否则直接输出规则结果
⑤ AI 解析 ai
查 LRU 缓存 → 构建 system+user prompt（强弱策略 + 上下文注入）→ instructor 结构化调用 DeepSeek → 计费统计 → 写缓存
⑥ 结果校验与合并
code/name 反查 → 直辖市映射 → 从属关系校验 → 与规则候选合并去弱 → AI 候选 confidence 取下限 60
⑦ 后处理与输出
残留行政后缀消费（自治县民族词）→ 重新计算 confidence → 按 (-confidence, -score) 排序 → 取前 max_candidates 个返回
