一切的起点

在 Duct Tape 之前，先有一把锤子——2024 年底那个只有 700 行的单文件解析器。

前身 address-py

2024 年 10 月 9 日，第一个 commit 推送到仓库。基于 jionlp 开源词典的思路自研，核心就两个文件： lcparser.py（~700行，LocationParser 解析器）+ address_api.py（~95行，FastAPI 服务封装）。

# address_api.py — 极简到骨子里 class Item(BaseModel): address: str @app.post("/analysis/") async def get_address_analysis(item: Item): old_dict = jio(item.address, town_village=True) # 从6位行政码截取省市代码，无AI无缓存无异步

技术栈：Python 3.11 + FastAPI 0.115 + Pydantic 2.9，无 AI、无缓存、无异步、无配置化。
数据基础：自带的 china_location.txt（jionlp 五级词典，约 3 万行，省/市/县/乡/村）+ china_location_change.txt（2018 年后县级及以上地名变更映射）。
匹配算法：全量线性扫描——对 3000+ 条省市区组合逐条用 in 操作符做子串包含判断，然后走 5 级过滤流水线。

算法 5 级过滤流水线
1 去重复 offset：同一位置匹配了省全名+市别名时丢弃别名项
↓
2 命中数量最多：优先省市区三级都命中的候选
↓
3 位置最靠前：sum(offset) 最小者优先
↓
4 全名优先于别名：有全名匹配时过滤纯别名匹配
↓
5 别名级别越高越好：如"西安交通大学"→西安市区非通辽市西安区

此外还有两个精巧的边界处理：别名后缀异常过滤（"太原路"不误匹配山西省太原）和 相邻 offset 防误判（"青海西宁"不误匹配"海西"）。 最后的 old2new_loc_map 还能把"襄樊市"自动转为"襄阳市"。

诞生的契机

两份 Excel 的对照，2836 个一致，153 个有风险——这就是一切的开始。

缘起 白泽(旧) vs 开放平台标准(新)

业务中发现两个系统的省市区代码不一致：旧系统"白泽"的代码表与新的"开放平台标准"代码表存在 153 个差异记录。 需要一个新的地址解析服务——以开放平台标准为唯一数据基准，智能解析用户输入的中文地址。

Excel 基准

核心需求：① 以开放平台 xlsx 为唯一基准 ② 返回候选集合供用户选择 ③ 返回省/市/区代码+名称+街道 ④ 智能解析（不只是正则） ⑤ 半年后可替换 xlsx 无需改代码。

四个关键战役

真实生产环境中的"破案"故事，每一次都让系统更健壮。

战役一 "元宝山区"截断案

地址"元宝山区珠江大道"被规则解析为上海市宝山区——因为"元宝山区"的后三个字恰好是"宝山区"。 规则自信地打 100 分，未触发 AI。

治法：截断检测（匹配名是更长区划名的后缀 → -30 分 + 强制 AI review）， 同时把截断候选"元宝山区(赤峰)"传入 AI 作为备选项。
修复后：AI 结合截断候选和 cityCode(150400 赤峰)，正确返回内蒙古赤峰市元宝山区。

战役二 "48单元"单字撞字案

地址"大池路督府天承18栋48单元502"——规则把"48单元"的"单"字匹配为"单县"简称， 由单县反推出山东省菏泽市，AI 又被"只能从候选选"的约束绑住无法纠正。

治法（v2.4）：弱猜测判定——省/市/县均无精确匹配时放开 AI 约束，AI 可基于常识自由判断。
修复后：AI 正确识别为江苏省无锡市梁溪区。
进一步优化（v2.5）：Prompt 增加区县推断规则，AI 主动填 county_name="滨湖区"，系统按名称反查补全代码。

战役三 "灵石路"模糊撞字案

地址"灵石路656号"——规则把"灵石路"模糊匹配为"灵石县"，AI 虽判断出上海静安区， 但返回的代码 310100 在数据集中查不到（数据集用 310100 表示上海市本身，静安区是 310106），导致正确结果被丢弃。

治法（v2.4）：AI 按名称反查区划代码（code 查不到时以 name 为准）+ 直辖市代码兼容映射（310000↔310100）。 弱猜测判定同样覆盖 fuzzy 撞字场景。

战役四 直辖市"缺区却 100 分"案

地址"江苏省无锡市大池路督府天承18栋502"——AI 返回 confidence=90、reason= "大池路位于滨湖区但未写明区名"，但经系统 merge+规范化后被规则覆盖为 confidence=100、 reason="全称匹配；省市层级完整；区县缺失；最终置信度100"，自相矛盾。

治法（v2.5）：置信度计算中增加 source="ai" 短路分支——AI 候选保留自身置信度和 reason， 不再被规则重算覆盖。同时 Prompt 增加区县推断规则，AI 主动填入 county。

接口完整文档

新老引擎字段逐一说明。建议新项目直接使用 v2.6 接口。

📥 请求参数（v2.6）
V2.6 POST http://localhost:8769/api/address/resolve
POST /api/address/resolve Content-Type: application/json { "address": "待解析地址", "useAI": true, "cityCode": "320200" }
字段类型必填默认值说明
address string✅— 待解析的中文地址。校验规则：至少含 6 个汉字（剔除数字/字母/标点后），且不能全为数字。不满足则 code="GO_BACK"，具体原因见同级 desc 字段。
useAI bool❌false 是否启用 AI 兜底。true/1=开启；false/0=关闭。建议生产环境开启，仅规则置信度低时触发，AI 调用率 <0.5%，单次成本约 ¥0.0004。
cityCode string❌"" 业务上下文城市编码（6 位）。提供后可大幅提升同名区县消歧准确率。例如只写了"滨湖区"，传入 cityCode="320200" 可锁定为无锡市滨湖区。强烈建议传入。
📤 响应字段（v2.6）
{ "code": "SUCCESS", "desc": "地址解析完成", "total": 1, "aiUsed": 0, "aiCacheHit": 0, "aiCost": { /* 见下方 */ }, "addressResolutionInfoList": [ /* 见下方 */ ] }
📋 顶层字段
字段类型说明
codestring 状态标识，二值。"SUCCESS"=解析成功（addressResolutionInfoList 非空）；"GO_BACK"=无有效结果（addressResolutionInfoList 为空，具体原因见同级 desc 字段）。
descstring 人类可读的描述。SUCCESS 时如 "地址解析完成"；GO_BACK 时如 "地址汉字数量不足，至少需要6个汉字，当前3个" 或 "AI兜底已尝试但无结果"。
totalint 候选结果数量（即 addressResolutionInfoList 数组长度，最大为 3）。
aiUsedint 是否触发了 AI 兜底。1=触发；0=未触发。可用于统计 AI 触发率。
aiCacheHitint AI 结果是否命中缓存。1=命中（零边际成本）；0=未命中。aiUsed=1 且 aiCacheHit=1 时，本次 AI 成本为零。
aiCostobject AI 费用明细（见下方 aiCost 表）。未触发 AI 时所有字段为零。
addressResolutionInfoListarray 候选结果数组。即使只有一个结果也放在数组里（与老接口最大结构差异）。按置信度降序排列，最多 3 个。
📋 aiCost 字段（费用可观测 · v2.6 峰谷双档）
字段类型说明
inputCacheHitTokensint命中缓存的输入 token 数。平时 ¥0.000014/个、高峰 ¥0.000028/个。数值越大成本越低。
inputCacheMissTokensint未命中缓存的输入 token 数。平时 ¥0.00014/个、高峰 ¥0.00028/个。主要是 Prompt 部分（固定约 116 token），每次必计费。
outputTokensintAI 输出 token 数。平时 ¥0.00028/个、高峰 ¥0.00056/个。一般为 40–150 token。
totalCostfloat本次 AI 调用的总费用（人民币元）。平时约 ¥0.0004、高峰约 ¥0.0008（不到 1 分钱）。
isPeakboolv2.6 新增。本次计费是否处于高峰时段（北京时间 9:00–12:00、14:00–18:00，单价约 2 倍）。未配置 peak_hours 时始终为 false。便于成本分析、对账与告警。
📋 addressResolutionInfoList[] 字段（候选结果）
字段类型说明
provinceCodestring省份代码，6 位，后 4 位 "0000"。例 "320000"=江苏省。与老接口格式一致。
provinceNamestring省份全称，如 "江苏省"。
cityCodestring城市代码，6 位，后 2 位 "00"。例 "320200"=无锡市。
cityNamestring城市全称。⚠️ 迁移注意：此处为驼峰 cityName（c 小写），老接口为 CityName（C 大写）。
countyCodestring区县代码，6 位完整代码。例 "320206"=惠山区。
countyNamestring区县全称。⚠️ 迁移注意：此处为驼峰 countyName（c 小写），老接口为 CountyName。
villagestring乡镇/村名。v2.6 暂未实现乡镇级解析，此字段始终为空。如需乡镇级可暂时使用老接口。
confidenceint置信度（0–100 分），调用方核心决策依据。≥90 可自动使用；75–90 建议人工审核；<75 建议拒绝。
计算：省份+30、城市精确+25、城市简称+18、区县精确+30、区县简称+20、cityCode强匹配+20、周边+10、AI最低60。
sourceint结果来源。0=规则引擎；1=AI 兜底。source=1 时建议结合 reason 审查。
reasonstring推理路径，人类可读。规则示例："省份Trie匹配+城市Trie匹配+区县Trie匹配"；AI 示例："AI判断：大池路位于滨湖区，结合cityCode推断为无锡市滨湖区"。可用于排查异常和审计。
✅ 新接口核心优势：confidence 置信度让调用方可决策；source+reason 让 AI 结果可解释；aiCost 让费用透明；返回数组支持多候选；入参 useAI/cityCode 提升准确率。
