### 0.2 前身核心实现思路（lcparser.py）

#### 0.2.1 数据组织：缩进式五级词典

`china_location.txt` 采用缩进制表符表达层级：
```
上海市	310000	上海              ← 省/直辖市（全名 + 行政码 + 别名）
	上海市	310100	上海          ← 市（缩进1个tab）
		嘉定区	310114	嘉定      ← 县/区（缩进2个tab）
			华亭镇                ← 乡镇/街道（缩进3个tab）
				北新村             ← 村/社区（缩进4个tab）
```
加载时按缩进层级解析为嵌套 dict，并构建扁平化的 `administrative_map_list`，每条记录为：
```
[admin_code, [省全名, 省别名], [市全名, 市别名], [县全名, 县别名], 是否最新地名]
```

#### 0.2.2 匹配算法：全量线性扫描 + 多维排序

`get_candidates(location_text)` 对 `administrative_map_list`（全国所有省市区组合，约 3000+ 条）做**线性全量遍历**：

1. **逐条比对**：对每条行政记录的省/市/县全名与别名，用 `in` 操作符判断是否出现在地址文本中
2. **记录命中**：统计命中数量 `count`，记录每个层级命中的 `offset`（在地址中的索引位置）和是否别名
3. **异常别名过滤**：处理"上海市嘉定区太原路"中"太原"被误匹配为山西省别名的问题（别名后紧跟"路/街/大街"时跳过）
4. **相邻 offset 防误判**：相邻层级命中位置相差 1 个字符时判定误匹配（如"青海西宁"误匹配"海西"）

#### 0.2.3 候选筛选：五级过滤流水线

得到候选列表后，依次按 5 个维度过滤/排序：

| 步骤 | 策略 | 说明 |
|------|------|------|
| 2.0 | 去重复 offset | 同一位置匹配了省全名+市别名时丢弃别名项（如"湖南省长沙市"误匹配长沙县） |
| 2.1 | 命中数量最多 | 优先选择省市区三级都命中的候选 |
| 2.2 | 位置最靠前 | `sum(offset)` 最小者优先，且省市区需按顺序排列 |
| 2.3 | 全名优先于别名 | 有全名匹配时过滤掉纯别名匹配（如"海南藏族自治州"不误匹配"海南省"） |
| 2.4 | 别名级别越高越好 | 全部为别名匹配时，优先匹配高级别（如"西安交通大学"匹配西安市而非通辽市西安区） |

#### 0.2.4 旧地名映射

`china_location_change.txt` 记录 2018 年后县级及以上地名变更（如襄樊市→襄阳市）。加载时融入 `administrative_map_list`（标记 `is_new=False`），并构建 `old2new_loc_map`，命中旧地名时可自动转为新地名+新代码。

#### 0.2.5 服务封装（address_api.py）

FastAPI 极简封装：
- 入参 `{address}`，调用 `LocationParser()(address, town_village=True)`
- 从 `areacode`（6 位行政码）截取前 2 位 + `0000` 作为省码、前 4 位 + `00` 作为市码
- 省市区任一为 None 时返回 `statueCode=201, code=ERROR`
- 无缓存、无 AI、无异步、无配置化

### 0.4 优劣对比

#### 0.4.1 前身（address-py）的优势

1. **支持五级行政区划**：可解析到乡镇/街道、村/社区，当前项目仅到区县
2. **内置旧地名映射**：`china_location_change.txt` 覆盖 2018 年后县级及以上地名变更（襄樊→襄阳、伊春美溪区→伊美区等），当前项目无此能力
3. **实现极简**：单文件 700 行，无外部依赖（除 FastAPI），部署运维成本低
4. **别名后缀异常处理**：对"太原路""重庆路"等别名+路名撞字有专门过滤逻辑

#### 0.4.2 前身（address-py）的劣势

1. **无 AI 兜底**：规则匹配不到时直接返回错误，无法处理"大池路督府天承"这类缺区但可推断的地址
2. **无置信度评估**：返回结果不告知可信度，调用方无法判断是否需要人工复核
3. **无缓存**：每次全量扫描，重复地址无法复用
4. **全量线性扫描性能差**：3000+ 条记录逐条 `in` 比对，`get_candidates` 是性能瓶颈（代码注释中明确标注"整个功能最耗时部分"）
5. **无配置化**：参数全硬编码，调参需改代码
6. **无异步**：高并发场景性能受限
7. **数据时效性依赖词典文件**：`china_location.txt` 需手动更新，且未与开放平台标准代码表对齐
8. **接口字段不规范**：`CityName`/`CountyName` 大小写不一致，无 `reason`/`confidence` 等可观测字段
9. **无费用可观测性**：不涉及 AI 调用，但若后续接入也无扩展基础

#### 0.4.3 当前项目（shangfenDH）的优势

1. **双 Trie 树索引**：前缀匹配 O(n)，性能远优于全量线性扫描
2. **AI 兜底 + 防幻觉**：规则置信度不足时自动触发 AI，结构化输出 + 名称反查 + 从属关系校验，准确率 99.96%
3. **置信度评估模型**：多维评估，调用方可按 `confidence` 决策是否人工复核
4. **cityCode 业务上下文消歧**：同名区县（西湖区/朝阳区/鼓楼区）可按业务上下文消歧，前身无法处理
5. **AI 缓存（TTL + LRU）**：重复地址命中缓存，成本与延迟双降
6. **全异步架构**：高并发友好
7. **配置化**：4 类 16+ 参数可配置，调参无需改代码
8. **费用可观测性**：每次 AI 调用的 token 用量与费用实时返回（v2.5）
9. **数据基准对齐开放平台**：省市区代码与名称完全对齐开放平台标准，避免代码不一致问题
10. **可观测性全面**：`reason` 推理理由 + `source` 来源标识 + `aiUsed/aiCacheHit/aiCost` 状态字段

#### 0.4.4 当前项目的劣势（相对前身）

1. **不支持乡镇/村**：业务暂不需要，若未来需要需扩展数据与索引
2. **无旧地名映射**：若遇到历史地址含旧地名（如"襄樊市"），当前项目依赖数据源时效性，无主动映射能力
3. **复杂度高**：AI + 缓存 + 异步 + 配置化，部署运维成本高于前身
4. **依赖外部 AI API**：网络/密钥/费用依赖，前身完全离线

### 0.5 演进动机总结

前身 `address-py` 验证了"基于行政区划词典的规则解析"思路的可行性，但在生产使用中暴露三大痛点：
1. **缺区/缺市时无法推断** → 当前项目引入 AI 兜底 + 区县推断规则
2. **同名区县无法消歧** → 当前项目引入 cityCode 业务上下文
3. **无置信度导致调用方无法决策** → 当前项目引入多维置信度评估模型

当前项目在前身的规则解析基础上，通过"双 Trie 树索引 + 置信度评估 + AI 兜底 + cityCode 消歧 + 缓存 + 配置化"六大能力升级，将准确率从约 99.7%（纯规则）提升至 99.96%（规则+cityCode+AI），同时实现了费用透明与全链路可观测。

## 第一部分：项目背景与目标（通用）

### 1.1 业务需求

将用户输入的中文地址文本（如"河南省郑州市中原区桐柏路123号"）解析为标准化的省市区代码及街道信息，并以 HTTP 接口形式提供服务。要求：

1. 以《区划代码（开放平台标准）.xlsx》为唯一数据基准，省市区代码必须对照该标准
2. 返回结果为集合（应对一个地址解析出多个相似候选，让用户选择）
3. 返回字段包含：省/市/区的代码与名称（单独字段）、中文街道字段
4. 解析要"智能"，不能仅靠正则匹配，需考虑各种区划类型（省/市/区/县/旗/自治旗/自治州/盟/地区/林区/特区/新区等）
5. 半年后国家更新省市区时，只需替换 xlsx 文件即可，无需改代码

### 1.2 技术选型

| 维度 | 选型 | 理由 |
|---|---|---|
| 主语言 | Python 3.12 | 数据处理与 NLP 生态成熟 |
| Web 框架 | FastAPI | 异步、自动文档、Pydantic 校验 |
| 数据结构 | Trie 树（自实现） | 区划名称前缀匹配，O(n) 复杂度 |
| 模糊匹配 | Levenshtein 距离 + 拼音 | 应对错别字与同音不同字 |
| AI 兜底 | openai 2.41.0 + Instructor + Pydantic | 结构化输出、防幻觉、自动重试 |
| AI 模型 | DeepSeek (deepseek-chat) | 成本低、中文能力强 |
| 数据读取 | pandas | Excel 读取 |

### 1.3 设计原则（贯穿所有版本）

- **规则优先，AI 兜底**：绝大部分地址由规则引擎秒级解析，仅低置信度地址触发 AI
- **数据驱动，零硬编码**：所有区划代码从 xlsx 动态加载，无任何硬编码区划代码
- **防幻觉**：AI 只能在给定候选范围内选择，返回后校验代码必须存在于标准库
- **多候选返回**：歧义地址返回多个候选，按可信评分排序，第一名尽可能准确

## 第二部分：v1.0 纯规则引擎

### 2.1 版本目标

建立基于 Trie 树的精确匹配引擎，覆盖全称、简称、别名三种匹配方式，辅以模糊匹配兜底，实现毫秒级地址解析。

### 2.2 核心设计

#### 2.2.1 双 Trie 树索引

构建两棵 Trie 树加速名称匹配：

- **full_trie**：插入所有区划**全称**（如"河南省""郑州市""中原区"）
- **short_trie**：插入所有区划**简称/别名**（如"河南""郑州""中原"）

`search_all(text)` 从文本每个位置开始前缀匹配，返回所有命中 `(start, end, word, [AreaInfo])`。

#### 2.2.2 分层匹配策略

每层匹配遵循"全称优先 + 简称补充 + 有效性过滤"：

```
省匹配 → 市匹配(受省约束) → 区县匹配(受市约束)
```

#### 2.2.3 简称/别名生成

```python
def get_aliases(name, level):

| city | 自治州、地区、盟、新区、市 |
| county | 自治旗、自治县、特区、林区、新区、区、县、市、旗 |

#### 2.2.4 模糊匹配兜底

当规则匹配无结果时，对文本分段做编辑距离+拼音模糊匹配：

```python
def _fuzzy_resolve(text):

## 第三部分：v1.1 AI兜底增强

### 3.1 演进动机

v1.0 的纯规则引擎存在 0.3% 的失败案例，主要集中在同名歧义、截断匹配、错别字等场景。这些场景规则难以覆盖，但 AI 大模型具备常识推理能力可以补充。核心问题是：**如何判断解析可能失败？何时触发 AI？**

### 3.2 新增能力

#### 3.2.1 置信度评估模型（解决"何时触发AI"）

设计 0-100 分的置信度评分体系，从多维度量化解析结果可信度：

```python
def _calc_confidence(cand, all_candidates, text):
    # 1. 匹配方式基础分
    sources = [province_source, city_source, county_source]
    if "fuzzy" in sources:        base = 60
    elif "infer" in sources:      base = 70
    elif "short" in sources:      base = 80
    else:                          base = 100  # 全称

    # 2. 层级完整度
    if 省+市+区县齐全:             level_bonus = 20
    elif 省+市:                    level_bonus = 10
    elif 省 or 市:                 level_bonus = 5

    # 3. 候选稳定性（多候选分差小=不稳定）
    if len(candidates)>=2 and top1-top2 < 15:  stability = -20
    else:                                        stability = 10

    # 4. village 异常惩罚
    if village含区划后缀字 or village首字是街道字:  village_penalty = -25

    # 5. 同名区县惩罚
    if county同名数量>1:            same_name_penalty = -15

    return clamp(base + level_bonus + stability + village_penalty
                 + same_name_penalty, 0, 100)
```

### 4.2 新增能力

#### 4.2.1 cityCode 业务上下文消歧（最重要的准确率提升）

**消歧策略分三层，逐层收紧**：

```python
def _apply_city_hint(self, candidates, city_code):
    same_province_codes = self.loader.get_same_province_city_codes(city_code)
    for c in candidates:
        if c.city.code == city_code:           # 强匹配：+20分
            c.confidence = min(100, c.confidence + 20)
        elif c.city.code in same_province_codes:  # 周边匹配：+10分
            c.confidence = min(100, c.confidence + 10)
```

1. **强匹配**：候选 city_code == 传入 cityCode → 置顶，置信度 +20
2. **周边匹配**：候选与传入 cityCode 同省 → 置顶，置信度 +10（应对跨市投保）
3. **AI 阶段注入**：把 cityCode 对应的省市信息作为强提示传给 AI

**关键设计**：cityCode 是"参考"而非"强制"，不直接过滤掉其他候选，而是调整排序。因为存在用户跨地市投保的真实场景。

        self.candidates = candidates          # 候选列表
        self.ai_used = ai_used                # AI 结果是否参与了候选
        self.ai_cache_hit = ai_cache_hit      # 是否命中 AI 缓存
        self.ai_attempted = ai_attempted      # AI 路径是否被尝试（区分"没调"和"调了但失败"）

    @property
    def empty(self):
        return not self.candidates
```

| 场景 | ai_used | ai_cache_hit | ai_attempted | desc 描述 |
|---|---|---|---|---|
| 未启用 useAI | 0 | 0 | 0 | 地址解析完成 |
| 启用 useAI，置信度高未触发 AI | 0 | 0 | 0 | 地址解析完成 |

### 5.8 Bug修复：AI模式直辖市场景重复候选

#### 5.8.1 问题复现

入参：
```json
{"address":"重庆市秀山县土家苗族自治县乌杨街道园区路21号", "useAI":1, "cityCode":""}
```

修复前返回（total=2，两条省市区编码完全相同）：
```json
{
  "addressResolutionInfoList": [
    {
      "provinceCode": "500000", "cityCode": "500100", "countyCode": "500241",
      "confidence": 85, "source": 0,
      "reason": "简称匹配；省市区层级完整；街道残留区划后缀；最终置信度85"
    },
    {
      "provinceCode": "500000", "cityCode": "500100", "countyCode": "500241",
      "confidence": 75, "source": 1,
      "reason": "简称+推断匹配；省市区层级完整；街道残留区划后缀；最终置信度75"
    }
  ],
  "total": 2, "aiUsed": 1
}
```

#### 5.8.2 根因分析

问题出在 `resolver.py` 的 `_merge_ai_candidates` 与 `_normalize_candidate` 的**执行顺序**。

`_merge_ai_candidates` 按 `(province.code, city.code, county.code)` 三元组去重，但它在 `_normalize_candidate`（直辖市标准化）**之前**执行。对于直辖市场景，标准化前后 key 会发生变化：

| 候选来源 | 标准化前 key | 标准化后 key |
|---|---|---|
| 规则候选 | `(500100, 500241, 0)` | `(500000, 500100, 500241)` |
| AI候选 | `(500100, 500100, 500241)` | `(500000, 500100, 500241)` |

合并时两者 key 不同（规则候选的 county 为 0，AI 候选的 county 为 500241）→ 都保留 → 标准化后两者变成相同的三元组 `(500000, 500100, 500241)`，但去重已经做完了，无法再合并 → 返回 2 条重复数据。

**关键链路**：
1. 规则匹配时，重庆是直辖市，`500100`（重庆市）level=province，`500241`（秀山县）level=city，规则候选的 county 为空
2. AI 返回的候选 province=500100、city=500100、county=500241（三层都有）
3. 合并时 key 不同 → 保留两条
4. 合并后才执行 `_normalize_candidate`：规则候选的 city(500241) 被赋给 county，province(500100) 被赋给 city，province 标准化为 500000
5. 此时两条数据省市区完全相同，但已无法去重

#### 5.8.3 修复方案

在调用 `_merge_ai_candidates` **之前**，先对 AI 候选和规则候选执行 `_normalize_candidate`，保证合并时 key 一致：

```python
# resolver.py - resolve() 和 resolve_async() 两处同步修改
if ai_cands:
    ai_used = True
    for ac in ai_cands:
        if ac.village is None:
            ac.village = self._extract_village(text, ac)
        ac.confidence = max(ac.confidence, 60.0)
        self._normalize_candidate(ac)          # 新增：先标准化 AI 候选
    for c in candidates:
        self._normalize_candidate(c)            # 新增：先标准化规则候选
    candidates = self._merge_ai_candidates(candidates, ai_cands)
    if city_code:
        self._apply_city_hint(candidates, city_code)
```

`_normalize_candidate` 是幂等操作（多次调用结果一致），合并前的标准化与合并后的再次标准化不会冲突，只是重复执行一次，开销可忽略。

## 第五部分补二：v2.4 AI弱猜测场景优化 + 名称反查

### 5.11 版本目标

v2.3 上线后，业务方反馈两个缺省市区信息的地址解析结果错误：

1. `大池路督府天承18栋48单元502` → 误返回山东省菏泽市单县（规则把「48**单**元」的「单」字误匹配为「单县」简称，AI 又被「只能从候选选」的约束绑住无法纠正）
2. `灵石路656号` → 误返回山西省晋中市灵石县（规则把「**灵石**路」的「灵石」模糊匹配为「灵石县」，AI 虽判断出上海静安区但返回结果被丢弃）

根因是：**当地址缺少省市区信息、规则仅凭单字简称撞字或 fuzzy 模糊撞字反推出省市区时，这种"弱猜测"候选本身就是错的，却仍被当作硬约束喂给 AI**，导致 AI 无法纠正。此外 AI 按国标返回的直辖市省级代码（如 310000）在数据集中查不到（数据集用 310100），导致正确的 AI 结果被丢弃。

v2.4 针对这些问题进行修复，**核心解析逻辑不变，仅优化 AI 兜底路径**。

### 5.12 弱猜测候选判定（核心改动）

#### 5.12.1 问题链路

以「大池路督府天承18栋48单元502」为例：

```
1. 规则解析：short_trie 把「48单元」的「单」(位置12) 误匹配为「单县」简称
   → county=单县(371722), province_source=infer, city_source=infer, county_source=short
2. 由单县反推出山东省(370000)、菏泽市(371700)
3. need_ai_review=True，触发 AI
4. AI prompt 约束「只能从候选列表中选择」→ 候选只有山东单县这一条
5. AI 被迫确认错误候选 → confidence 被拉到 100
6. 返回错误的山东菏泽单县
```

#### 5.12.2 修复方案：弱猜测判定 + 放开 AI 约束

新增 `_is_weak_rule_candidate` 方法，判定规则候选是否为「弱猜测」：

```python
@staticmethod
def _is_weak_rule_candidate(cand):
    """省/市/县任一级在原文中精确匹配（full/short）即视为有可信锚点，非弱猜测。
    仅当三级均未精确匹配（纯靠 infer/fuzzy 撞字反推）时才判为弱猜测。"""
    if cand is None:
        return True
    if getattr(cand, "source", "") == "ai":
        return False
    precise = {"full", "short"}
    if (cand.province_source in precise
            or cand.city_source in precise
            or cand.county_source in precise):
        return False
    return True
```

**关键设计**：判定逻辑覆盖三种撞字场景：
- 单字简称撞字（infer）：如「单元」的「单」→ 单县
- fuzzy 模糊撞字：如「灵石路」的「灵石」→ 灵石县
- 截断匹配：省市县均无精确锚点

只要省、市、县**任一级**有 full/short 精确匹配（如「西湖区」全称匹配），就视为有可信锚点，不判为弱猜测——这保证同名歧义场景（如西湖区杭州 vs 南昌）仍走正常 AI 约束路径。

弱猜测场景下的 prompt 改造：

```
# 非 weak（正常场景）
1. 只能从候选列表中选择省市区代码，严禁编造不在候选中的代码。

# weak（弱猜测场景）
1. 规则引擎给出的候选仅为参考，可能存在单字简称或模糊撞字误匹配
   （如「48单元」的「单」误匹配「单县」、「灵石路」的「灵石」误匹配「灵石县」）。
   若候选与地址明显不符，应拒绝候选，基于常识从全国真实行政区划中选择。
   重要：不确定6位区划代码时，province_name/city_name/county_name 必须填写真实名称，
   对应的 code 字段可留 null，系统会按名称补全代码。
```

#### 5.12.3 丢弃乱猜候选

AI 返回结果后，丢弃规则的弱猜测候选，避免污染输出：

```python
def _merge_ai_candidates(self, rule_cands, ai_cands):
    merged = {}
    for c in ai_cands:
        # AI 候选先入 merged
        ...
    drop_weak = bool(merged)  # AI 有结果时才丢弃
    for c in rule_cands:

            continue  # 丢弃弱猜测候选
        ...
```

### 5.13 AI 按名称反查区划代码

#### 5.13.1 问题：AI 知道名称但瞎填代码

弱猜测场景放开约束后，AI 能基于常识判断出「上海市静安区」，但 AI 不确定静安区的 6 位代码（310106），会瞎填一个看似合理的代码（如填了 310100 上海市本身）。原 `_to_candidate` 只按 code 查，查到「上海市」就当 city，静安区丢失。

#### 5.13.2 修复：code 查不到或名称不符时按 name 反查

在 `data_loader.py` 新增 `find_by_name` 方法，支持按名称+层级+上级代码反查：

```python
def find_by_name(self, name, level=None, province_code=0, city_code=0):
    """按名称反查区划信息。可选限定 level / 上级代码以消歧。"""
    pools = []
    if level is None or level == "province":
        pools.append(self.provinces)
    if level is None or level == "city":
        pools.append(self.all_cities)
    if level is None or level == "county":
        pools.append(self.all_counties)
    for pool in pools:
        for info in pool:
            if info.name != name:
                continue
            if level and info.level != level:
                continue
            if province_code and info.province_code != province_code:
                continue
            if city_code and info.city_code != city_code:
                continue
            return info
    return None
```

`_to_candidate` 在 code 查不到、或查到的名称与 AI 给的名称不符时，以 name 为准反查：

```python
# code 查到的名称与 AI 给的名称不符时，以 name 为准反查
if item.province_name and (province is None or province.name != item.province_name):
    province_by_name = loader.find_by_name(item.province_name, level="province")
    if province_by_name is not None:
        province = province_by_name

if item.city_name and (city is None or city.name != item.city_name):
    prov_code_for_city = province.code if province and province.level == "province" else ...
    city_by_name = loader.find_by_name(item.city_name, level="city", province_code=prov_code_for_city)
    if city_by_name is not None:
        city = city_by_name

if item.county_name and (county is None or county.name != item.county_name):
    county_by_name = loader.find_by_name(item.county_name, level="county", city_code=city.code)
    if county_by_name is not None:
        county = county_by_name
```

#### 5.13.3 直辖市代码兼容

数据集里直辖市的省级代码是 `xx0100`（如 310100），而国标省级码是 `xx0000`（如 310000）。AI 按国标返回 310000 时数据集查不到，做反向兼容映射：

```python
_MUNI_PROV_AI_COMPAT = {110000: 110100, 310000: 310100, 120000: 120100, 500000: 500100}

province = loader.get_by_code(prov_code) if prov_code else None
if province is None and prov_code in self._MUNI_PROV_AI_COMPAT:
    province = loader.get_by_code(self._MUNI_PROV_AI_COMPAT[prov_code])
```

#### 5.13.4 从属关系校验（防张冠李戴）

AI 可能给出省和市不匹配的错误组合（如 province_name='浙江省' + city_name='南昌市'）。在 name 反查后增加从属关系校验：

```python
if province and city:
    prov_code_for_city = province.code if province.level == "province" else province.parent_code
    if city.province_code != prov_code_for_city:
        # 省市不匹配，以 city 为准反推 province
        real_prov = loader.get_by_code(city.province_code)
        province = real_prov if real_prov else province
if city and county:
    if county.city_code != city.code:
        # 市县不匹配，以 county 为准反推 city
        real_city = loader.get_by_code(county.city_code)
        if real_city:
            city = real_city
            real_prov = loader.get_by_code(city.province_code)
            if real_prov:
                province = real_prov
```

### 5.14 Prompt 直辖市数据模型说明

原 prompt 第 2 条「直辖市的区是市级，省级为直辖市本身」表述模糊，AI 理解成「区=city 字段，但直辖市无地级市」就全留空。v2.4 明确数据模型映射：

```
2. 直辖市(北京/上海/天津/重庆)数据模型：province 填直辖市本身(如'上海市')，
   其下辖的区(如'静安区''浦东新区')填到 city 字段，county 字段留 null。
   系统会自动把区归到 county 层级输出。即对'上海静安区'应填：
   province_name='上海市'、city_name='静安区'、county=null。
```

### 5.15 测试校验逻辑优化

原 `full_verify.py` 的 `verify_one` 只校验第一名（`cands[0]`），但同名歧义场景返回多候选（如西湖区同时返回杭州+南昌）是合理的，第一名未必是期望城市。v2.4 调整为：**期望的省市区在候选列表中即算通过**，街道校验仅对匹配到期望省市区的那个候选进行。

```python
# 旧逻辑：只校验第一名
c = cands[0]
if got_prov != exp_prov: return False

# 新逻辑：期望省市区在候选列表中即可
matched = None
for c in cands:
    if (c.province.code == exp_prov
            and (exp_city is None or c.city.code == exp_city)
            and (exp_county is None or c.county.code == exp_county)):
        matched = c
        break
if matched is None: return False
```

## 第五部分补三：v2.5 AI置信度修复 + 推断区县 + 费用可观测性

### 5.18 版本目标

解决两个生产问题：
1. **AI 候选置信度被规则误覆盖**：AI 候选经 merge 后被强制走规则置信度计算，`source="ai"` 无短路保护，导致 AI 自身的置信度与 reason 被整体覆盖，出现"缺区却 100 分、reason 自相矛盾"
2. **AI 不主动推断缺区**：prompt 未明确要求推断区县，AI 知道是滨湖区但保守留空
3. **费用黑盒**：无法观测每次 AI 调用的 token 用量与实际花费

### 5.19 AI 候选置信度短路保护（resolver.py）

#### 5.19.1 问题复现

入参 `江苏省无锡市大池路督府天承18栋502`（缺区），AI 返回：
```
confidence: 90, reason: "大池路位于滨湖区但未写明区名"
```

经系统 merge + 规范化后输出：
```
confidence: 100, reason: "全称匹配；省市层级完整；区县缺失；最终置信度100"
county: null
```

AI 自身的 90 分和 reason 被整体覆盖，且被规则当成"全称匹配"打 100 分底分，最终 clamp 到 100。

#### 5.19.2 根因

`_calc_confidence` 中只有 `source="guided"` 的候选有短路保护，`source="ai"` 的候选落入 `else: base=100` 分支被当成"全称匹配"打 100 分底分。

#### 5.19.3 修复方案

在 `_calc_confidence` 中增加 `source="ai"` 短路分支：AI 候选保留自身置信度和 reason，仅做轻微稳定 +10，不再被规则重算覆盖。与 `guided` 候选同等的保护方式。

### 5.20 AI Prompt 增加区县推断规则（ai_resolver.py）

#### 5.20.1 问题

AI 知道"督府天承"位于滨湖区，但 prompt 未要求推断区县，AI 出于保守把 county 留空。

#### 5.20.2 修复

在 system prompt 增加第 7 条「区县推断规则」：
> 地址明确写出省市但缺区县时，若能根据街道/路名/小区名高置信度唯一确定所属区县，必须填入 `county_name`（code 不确定可留 null，系统按名称补全）

修复后 AI 主动填 `county_name="滨湖区"`，系统按名称反查补 `county_code="320211"`。

### 5.21 直辖市规范化 reason 同步（resolver.py）

#### 5.21.1 问题

直辖市场景下 AI 按 prompt 把区填到 `city` 字段、reason 写"county留null"，系统规范化把区从 `city` 提升到 `county`，但 reason 未同步修正，导致 reason 与实际输出矛盾。

### 5.26 配置扩展（config.ini + config.py）

### 5.27 计费逻辑改造（ai_resolver.py）

#### 5.27.1 高峰时段解析

新增 `_parse_peak_hours(peak_hours)` 静态方法，将 `"09:00-12:00;14:00-18:00"` 解析为分钟区间列表 `[(540, 720), (840, 1080)]`（左闭右开）：
- 多段用分号分隔
- 非法格式（如 `abc`、`25:00-26:00`）整段跳过
- 跨日区间（如 `23:00-06:00`）跳过
- 空字符串返回 `[]`，等价于不开峰谷

新增 `_is_peak_now()`：以北京时间（UTC+8）取当前时分，判断是否落在任一高峰区间内。`peak_ranges` 为空时直接返回 `False`。

#### 5.27.2 峰谷单价选择

`_build_cost_info(usage)` 在计算 `totalCost` 前先调 `_is_peak_now()`，按结果选用对应档位单价：

```python
is_peak = self._is_peak_now()
if is_peak:
    p_hit, p_miss, p_out = self.price_peak_input_cache_hit, self.price_peak_input_cache_miss, self.price_peak_output
else:
    p_hit, p_miss, p_out = self.price_input_cache_hit, self.price_input_cache_miss, self.price_output
total_cost = (cache_hit * p_hit + cache_miss * p_miss + completion_tokens * p_out) / 1_000_000.0
```

返回 dict 新增 `isPeak` 字段（bool），标识本次计费是否按高峰单价计算。

#### 5.27.3 向后兼容

构造函数新增 4 个参数 `price_peak_*` 与 `peak_hours`，全部带默认值：
- `peak_hours` 默认 `""` → `peak_ranges=[]` → 永远走平时单价
- `price_peak_*` 默认 `None` → 回退为对应平时单价

因此**遇到普通模型（如 deepseek-chat 或其他厂商）时，只需留空 `peak_hours` 并只填平时单价**即可，无需改动代码。

### 5.28 透传与响应结构（resolver.py + server.py）

- `ResolveResult.ai_cost_info` 默认结构与 `_zero_cost()` 均新增 `isPeak: False` 字段，保持未调用 AI 时响应结构一致

### 5.4 核心解析引擎

#### 5.4.1 主流程（v2.0 异步版）

```python
async def resolve_async(self, address, max_candidates=3, use_ai=False, city_code=None):
    text = _preprocess(address)
    candidates = await asyncio.to_thread(self._rule_resolve, text)  # CPU密集→线程池

    if city_code:
        self._apply_city_hint(candidates, city_code)  # cityCode消歧

    if use_ai and self.ai_resolver and self._need_ai(candidates):
        ai_cands = await self.ai_resolver.resolve_async(address, text, candidates, self.loader, city_code=city_code)
        if ai_cands:
            candidates = self._merge_ai_candidates(candidates, ai_cands)
            if city_code:
                self._apply_city_hint(candidates, city_code)  # AI结果二次消歧

    candidates.sort(key=lambda c: (-c.confidence, -c.score))
    return candidates[:max_candidates]
```

#### 5.4.2 简称有效性过滤（核心难点）

简称匹配容易误伤，设计了多重过滤：

**`_is_short_invalid`：后缀字冲突过滤**

当简称后紧跟区划后缀字（区/县/市/旗/省/州/盟）时，需判断该后缀字是否是更长区划名的一部分：

```python
def _is_short_invalid(text, start, end, word):

        rest = text[end:]
        # 检查剩余文本是否能组成区县/市全称或简称

**`_is_county_short_street_conflict`：街道后缀冲突过滤**

```python

#### 5.4.3 候选组合 `_combine`

按"省 → 市 → 区县"的层级关系组合候选，处理多种情况：

```

│      ├─ 是 → 该市下有区县匹配?
│      │      ├─ 是 → 省+市+区县
│      │      └─ 否 → 检查市全称与区县是否重叠?
│      │           ├─ 重叠 → 回退用市简称让位给区县 (_find_short_city)
│      │           └─ 不重叠 → 省+市
│      └─ 否 → 检查省与市是否位置重叠且省份不一致?

├─ 否(有市) → 市+区县 或 仅市，省由市推断

### 5.7 多候选返回机制

**核心机制**：

```python
candidates.sort(key=lambda c: (-c.confidence, -c.score))  # 按置信度降序
return candidates[:max_candidates]  # max_candidates默认3，最多返回3个
```

- **最多返回 3 个候选**
- **按置信度降序排列**，第一名置信度最高
- **是否返回多个取决于是否存在歧义**

**4种场景的实际行为**：

| 场景 | 返回数量 | 第一名 |
|---|---|---|
| 无歧义（唯一确定） | 1个 | 唯一正确结果，conf=100 |
| 同名歧义（无cityCode） | 多个 | 按规则置信度排序 |
| 同名歧义 + cityCode | 多个 | 投保地市候选置顶，conf+20 |
| 截断匹配 + AI兜底 | 多个 | AI修正结果置顶，conf=95 |

**关键设计**：cityCode/AI 只调整排序，不删除候选。消歧不是"过滤"，而是"排序"，保留其他可能性给上游业务做二次决策。

## 第七部分：关键技术决策与权衡

### 6.1 为什么用 Trie 而非正则？

- 正则无法高效处理"前缀匹配+多候选"，Trie 的 `search_all` 一次扫描返回所有命中
- Trie 支持动态插入，xlsx 更新后重建索引即可
- 复杂度 O(n×L)，n为文本长度，L为最长区划名长度

### 6.2 为什么不全用 AI？

- **成本**：1万条地址全用 AI 约需 1万次 API 调用，规则引擎免费
- **延迟**：规则解析毫秒级，AI 调用秒级
- **可控性**：规则结果可解释、可复现，AI 有幻觉风险
- **准确率**：规则已达 99.77%，AI 仅补充边缘 case

### 6.3 为什么 AI 只能选不能生成？

- AI 自由生成代码会编造不存在的区划（幻觉）
- 约束在候选范围内选择，准确率从"可能错"变为"必在候选中"
- 返回后二次校验代码存在性，双重防幻觉

### 6.4 置信度阈值为何选 75？

- 经测试，75 分能覆盖大部分歧义场景（同名、截断、单字简称）
- 过低（60）会漏判，过高（85）会过度调用 AI 增加成本
- 配合 `need_ai_review` 标志，对截断/单字简称等明确风险强制触发

### 6.5 cityCode 为何是"参考"而非"强制"？

- 保险业务存在跨地市投保的真实场景（如投保杭州但地址写绍兴）
- 强制过滤会丢失这些合法候选
- 调整排序（+20/+10分）既能让投保地市置顶，又保留其他候选供二次决策

### 6.6 缓存 key 为何包含 city_code？

- 同一地址在不同 cityCode 下结果可能不同（消歧方向不同）
- 如"西湖区"在杭州cityCode下第一名是杭州西湖区，在南昌cityCode下第一名是南昌西湖区
- 缓存 key 必须包含 city_code 才能正确区分

### 6.7 直辖市层级为何特殊处理？

开放平台标准中直辖市的区是市级代码（如110105朝阳区是city），这与传统理解不同。系统未做特殊硬编码，而是完全遵循 xlsx 的 parent_code 层级关系，确保数据更新后自动适配。

### v2.1 业务价值

1. **可观测性**：业务方每次调用都能看到"本次是否走了AI、是否命中缓存、置信度多少"
2. **可解释性**：每条候选都附带推理理由（如"候选差异小存在歧义；同名区县(2个)"），业务方可理解"为什么解析成这个结果"
3. **可调试性**：失败时 `desc` 清晰区分"无法解析" / "AI兜底已尝试但无结果" / "地址不能为空"
4. **接口一致性**：状态码统一为二值（`SUCCESS` / `GO_BACK`），布尔字段统一为 0/1，与国内常见 RPC 协议一致

### v2.2 业务价值

1. **数据正确性**：消除 AI 模式下直辖市场景的重复候选返回，避免业务方收到冗余数据做二次决策
2. **工程可维护性**：核心代码、数据、测试、文档分层清晰，降低后续维护与协作成本
3. **向后兼容**：接口契约不变，准确率不变，现有调用方无需改动

### v2.4 业务价值

1. **消除弱猜测污染**：地址缺省市区时，规则的单字简称撞字和 fuzzy 撞字不再绑架 AI，AI 可基于常识自由判断
2. **AI 名称反查兜底**：AI 知道名称但不确定代码时，系统按名称反查真实区划代码，不再因代码填错而丢弃正确结果
3. **直辖市代码兼容**：AI 按国标返回的省级代码自动兼容数据集的直辖市码
4. **从属关系校验**：防止 AI 给出省和市张冠李戴的错误组合
5. **测试校验更合理**：同名歧义场景返回多候选时，期望值在候选中即算通过，准确率提升至 99.96%
